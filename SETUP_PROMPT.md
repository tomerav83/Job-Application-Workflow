- Install only after the user approves — nothing installed mid-step by surprise.

<step id="1" name="collect_base_cv">
Goal: get the base CV (HTML) — the source of truth for all work history and achievements.

1. Run `ls private-files/*-CV.html`. Convention: {First-Last}-CV.html, e.g. Jane-Doe-CV.html.
2. Branch on the result:
  - One match -> use it. Read it. Derive YOUR_NAME_SLUG from the filename by dropping
    "-CV.html" and keeping its casing (Jane-Doe-CV.html -> "Jane-Doe"). Record
    YOUR_BASE_CV_PATH = its absolute path (e.g. {repo}/private-files/Jane-Doe-CV.html). Tell the
    user, then skip to Step 2.
  - Several matches -> list them, ask which to use.
  - No match -> ask the user for their CV (step 4 below).
3. Keep the slug's casing: render-cv.js names its HTML/PDF output from the full name (e.g.
   "Jane-Doe.pdf"), so a Title-Case slug keeps the base CV, prompt paths, and rendered files
   consistent.
4. When no CV is pre-placed, ask for it (and note they can pre-drop it next time as
   private-files/{First-Last}-CV.html):
  - HTML -> ask for its path.
  - PDF/DOCX -> offer to convert to HTML, then read.
  - Pasted text -> capture it (you save it as HTML in Step 12).
5. Continue only once you have the CV content.
   </step>

<step id="2" name="provision_linkedin_mcp">
Goal: make the LinkedIn MCP server's tools available so Step 3 can pull the user's profile
directly. This is the ONLY LinkedIn source — there is no PDF export path anymore.

The server is `linkedin-scraper-mcp` (repo: stickerdaniel/linkedin-mcp-server). It scrapes the
authenticated user's own LinkedIn through a real browser session and exposes tools such as
`get_my_profile` and `get_person_profile`.

The ONLY thing the user installs is `uv`. Everything else uv handles on its own and you must
say so plainly, because it is non-obvious:
  - Python 3.12+ is NOT installed by the user and does NOT go on their system. The package needs
    3.12+, but `uvx` downloads its own private copy of Python into uv's cache
    (~/.local/share/uv/python/). It is not on PATH and does not touch this repo's Python 3.11.
  - The MCP server package and its Chromium (via Patchright) are likewise fetched automatically.

1. Check whether the MCP tools are ALREADY available in this session (look for a tool named
   like `mcp__linkedin__get_my_profile`). If present -> skip to step 5 (display preflight, then auth).
2. Per <dependency_protocol>, install uv only after approval:
   - uv (>= 0.4): `curl -LsSf https://astral.sh/uv/install.sh | sh` (skip if `uv --version` works).
   - Reassure the user: no separate Python install, no system change beyond adding uv.
3. Register the server with Claude Code at LOCAL scope so it is not committed:
     claude mcp add linkedin --scope local -e UV_HTTP_TIMEOUT=300 -- uvx linkedin-scraper-mcp@latest
   (Equivalent: a gitignored `.mcp.json` with the same command/args/env.)
4. MCP tools only load on a fresh session. Tell the user to reload/restart the session and
   re-invoke this setup, then STOP. On re-entry the earlier steps are already done (CV read,
   uv installed) and the `mcp__linkedin__*` tools will be present — continue from here.
5. DISPLAY PREFLIGHT (do this BEFORE any login attempt — the `--login`/`get_my_profile` browser
   is HEADED, so a working, VISIBLE display is required). The common failure is a window that
   loads but never appears on screen, which looks like a hang and wastes a lot of time; catch it
   up front:
   - If there is no display at all ($DISPLAY unset, true headless): there is no headed-login path
     here — go straight to the no-display fallback below (login on another machine, copy
     ~/.linkedin-mcp over) or skip LinkedIn.
   - If a display exists (e.g. WSLg sets DISPLAY=:0): do NOT trust `xset q`/`xdpyinfo` — those
     only prove the X socket answers, not that windows are painted. Actually paint one: run a
     trivial GUI app (`xmessage hi` or `xeyes`) and ASK the user whether it appeared.
       - Appears -> display works; proceed to login (step 6).
       - Does NOT appear (but X answers) -> the compositor is not exporting windows. On WSLg this
         is the RAIL bridge breaking; every browser launch will be invisible no matter what.
         Tell the user to FIX IT FIRST: from Windows PowerShell run `wsl --shutdown`, wait ~10s,
         reopen the WSL terminal (run `wsl --update` first if it persists), then re-invoke setup.
         Do NOT debug the browser, kill/relaunch Chromium, clear Singleton locks, toggle
         --disable-gpu, or unset WAYLAND_DISPLAY — none of those fix an unpainted compositor.
6. Authenticate once (browser session stored at ~/.linkedin-mcp/profile/):
   - Call `get_my_profile`. If it returns data -> already authenticated, move on.
   - Else run `uvx linkedin-scraper-mcp@latest --login` (opens a browser for manual sign-in;
     handles 2FA/captcha, ~5 min). Tell the user two things up front:
       (a) Sign in with LinkedIn EMAIL + PASSWORD, not "Sign in with Google" — Google blocks its
           OAuth flow inside automation-controlled browsers, so SSO gets stuck. If the account
           only has Google sign-in, set a LinkedIn password first (Forgot password) and use that.
       (b) Launch the login EXACTLY ONCE and wait. Repeated launches leave orphaned Chromium
           holding the profile, which causes "Opening in existing browser session" and a window
           that never shows. If a retry is truly needed, first kill stray chromium
           (`pkill -f chrome-linux64`) and clear `~/.linkedin-mcp/profile/Singleton*`, then
           launch once.
   - No-display / headless fallback: run `--login` on a machine with a browser and copy
     ~/.linkedin-mcp over, or use `--no-headless` under a working WSLg (see the preflight above).
7. Before the first scrape, get the user's explicit acknowledgement of the risk: LinkedIn's ToS
   prohibits automation; for personal, non-bulk use the project reports no known bans, but the
   account risk is the user's to accept.

If the user declines the uv/MCP install or cannot authenticate, there is NO PDF fallback. Tell
them which LinkedIn-sourced fields you cannot auto-fill (location, headline, summary, experience,
education, skills, languages, honors) and collect them directly from the user in Step 5.
   </step>

<step id="3" name="collect_linkedin">
Goal: pull the user's LinkedIn profile through the MCP (Step 2) — it fills location, headline,
summary, experience, education, skills, languages, and honors as STRUCTURED data (no PDF to parse).

1. Call `get_my_profile` (or `get_person_profile` on the user's own profile URL), requesting the
   experience, education, skills, languages, and honors sections.
2. If the call errors with a setup/session message, retry once after a short wait (cold Chromium
   start is normal on the first call). If it still fails, treat auth as missing and return to
   Step 2.6 (`--login`).
3. Map the returned structured fields to the Step 4 list. Note: the scraper returns exactly what
   is on the profile, so dates are only as granular as the user entered them — year-only role
   boundaries remain MISSING and are collected in Step 5 (never guessed). The MCP removes the
   lossy PDF parsing, not the gaps in the underlying profile.
4. Capture the user's OWN LinkedIn username/handle. Derive it from the `url` field in the
   profile result: `get_my_profile` navigates `/in/me/` and resolves the redirect to the REAL
   username (e.g. `https://www.linkedin.com/in/jane-doe/`). NEVER hardcode a username; always use
   the relevant (authenticated) user's resolved handle. Keep the public address for the CV header
   too (just the address, e.g. linkedin.com/in/jane-doe).
5. PULL THE FULL PER-ROLE SKILLS — the MCP truncates them. `get_my_profile`/`get_person_profile`
   return only the top ~10 skills with role tags and collapse each role's list as "+N skills",
   regardless of `max_scrolls`; there is no `detail`/`full` param and no flat "all skills" mode
   (confirmed against the installed source and the upstream repo). Get the complete list directly
   with the committed helper, which reuses the already-authenticated session:
   a. Close the MCP browser first so the session profile is free: call the `close_session` tool.
   b. Run, passing the resolved handle from item 4 (NEVER a hardcoded name):
        node scripts/linkedin-skills/scrape-linkedin-skills.js {USERNAME} --out /tmp/li-skills.json
      The script copies the session profile (leaving the MCP's original untouched), drives this
      repo's bundled Playwright chromium, iterates the skill category tabs (localized-UI safe),
      cleans up its temp copy, and writes clean {skill, attribution} JSON — where `attribution`
      is the per-role mapping used in Step 10 (e.g. "Release Engineer at Taboola", or "N
      experiences at Taboola and M other companies"). See scripts/linkedin-skills/README.md.
   c. If the script exits 2 (auth wall), the stored session expired — re-login via Step 2.6 and
      retry. If Playwright or the session is unavailable, fall back to collecting per-role skills
      from the CV + the user in Step 10 — never guess.
   </step>

<step id="4" name="extract_and_confirm">
Extract every field below from the CV (Step 1) + LinkedIn (Step 3). Present them as one filled
list to confirm, tagging each value [from CV] / [from LinkedIn] / [derived], and marking
anything not found as MISSING.

Personal (CV header):
- Full name — CV / LinkedIn
- Email — LinkedIn (top) or CV
- Phone — usually CV only (LinkedIn omits it), so often MISSING
- Location (city, country) — CV / LinkedIn
- LinkedIn URL/handle — Step 3

Professional profile (tailoring prompts):
- Current title — the latest role title exactly as written on the CV
- Title variants — the remaining CV role titles, each normalized to an "Engineer" suffix with
  the standard acronym where one exists (e.g. "Backend Software Developer" -> "Backend
  Engineer"; "Site Reliability Engineering Software Developer" -> "Site Reliability Engineer /
  SRE"). Confirm.
- Total years of experience — derive from earliest Experience start to today, rounded to a clean
  whole/half (e.g. 7 or 6.5); if dates are too vague, ask.
- Career start month/year — earliest Experience start date

CV content (resume template):
- Up to 3 personal achievements — LinkedIn "Honors & Awards" or CV awards (not work bullets); confirm
- Languages + proficiency — LinkedIn "Languages" / CV
- Up to 4 interests/hobbies — not on LinkedIn, so usually MISSING
- Job history — title, company, start/end dates for EACH role as its own entry (Experience /
  CV). Never merge roles: two roles at the same employer (e.g. a promotion or role change) are
  two separate entries, never one. Capture dates at MM/YYYY granularity where the sources give
  it; flag any role-boundary month that is not stated as MISSING. Keep roles distinct in every
  downstream artifact (Steps 9 and 10).

Then collect every correction or MISSING value via AskUserQuestion (<info_collection_protocol>)
before continuing.
</step>

<step id="10" name="create_candidate_context">
1. Check tailoring/candidate_context.md: real content vs empty/templated.
2. Real content -> copy to ~/Documents/job-application-automation/candidate_context.md.
3. Empty/templated -> build it with the user. Pre-fill from collected data (the full per-role
   skills scraped in Step 3.5 + LinkedIn Honors + CV), then have them review and add what is not
   on LinkedIn. Include:
   - Full skills and tech stack (everything they can speak to)
   - Work-history details and achievements not in the base CV
   - Adjacent skills — close enough to discuss in an interview (label as the
     <skills_adjacency_list> section)
   - Any domain experience (industries, product types, scale)
   - One <role_*> section PER ROLE from the Step 4 job history — each distinct role is its own
     section. NEVER merge multiple roles into one section, even when they share an employer (two
     roles at the same company = two separate sections).
   - Each <role_*> section's tech stack (languages, databases, tools) is INDEPENDENT and
     role-specific. Build it from THAT role's own scraped per-role skills (Step 3.5) plus that
     role's CV bullets, and confirm it with the user for that role ALONE. NEVER apply one role's
     answer to another role, and never ask a single question that sets the stack for multiple
     roles at once (e.g. do not ask one "Navy languages" question and write the answer into both
     Navy role sections — ask per role).
   - Name languages and technologies CONCISELY and CONCRETELY (e.g. "Java, C++"). NEVER use vague
     umbrella framings like "JVM-based services", "scripting languages", or "observability stack"
     in place of the actual names. If a role's concrete stack is unknown, ask the user for that
     role - never paper over it with an umbrella term.
   - Every role section's dates MUST use MM/YYYY format (e.g. "10/2019 - 03/2023"; the current
     role ends in "present"). If the month for any role boundary cannot be retrieved from the CV
     or LinkedIn, ask the user for it via AskUserQuestion before writing the file — never fall
     back to year-only and never guess.
4. For any section not grounded in the CV/LinkedIn (role product/industry/tech, general framing,
   off-limits), draft it instead of leaving a TODO:
   a. Research it — WebSearch/WebFetch the company/role for industry, product, and known stack;
      combine with the CV bullets. For non-public employers (e.g. classified/defense), infer
      conservatively from CV bullets and skills only.
   b. Present the draft via AskUserQuestion as the recommended (first) option (accept /
      alternative / Other to edit). Bundle related sections.
   c. Save only what the user confirms — no unverified guesses, no TODO left behind.
5. Save to both: tailoring/candidate_context.md and
   ~/Documents/job-application-automation/candidate_context.md.
</step>

<step id="11" name="personalize_prompts">

The personalized prompts go into a gitignored, timestamped folder instead.

1. Create the output dir:
   private-files/{timestamp}-generated-prompts/
   where {timestamp} is a human-readable local timestamp, e.g. 2026-05-27_14-30-05.
2. Copy both templates into that dir (keeping their filenames), then apply these replacements to
   the COPIES only:
  - YOUR_NAME -> full name
  - YOUR_NAME_SLUG -> First-Last with hyphens, matching the CV filename casing (e.g. "Jane-Doe");
    must match render-cv.js output naming (it names files from the full name)
  - YOUR_TITLE -> latest CV role title (Step 4)
  - YOUR_YEARS_EXP -> total years (number only)
  - YOUR_EXPERIENCE_START -> career start (e.g. "March 2018")
  - YOUR_LOCATION -> location
  - YOUR_TITLE_VARIANT_1 / _2 -> title variants (Step 4)
  - YOUR_YEARS_EXP+1 / +2 / +3 -> the actual computed numbers
  - ~/path/to/job-application-automation -> repo path (Step 6)
3. Confirm no YOUR_* or ~/path/to/* placeholders remain in the two generated copies, and that
   tailoring/TAILOR_PROMPT.txt and tailoring/COWORK_PROMPT.txt are still UNCHANGED (git clean).

The user pastes the prompts from private-files/{timestamp}-generated-prompts/ into Cowork;
the repo templates stay pristine.
</step>
