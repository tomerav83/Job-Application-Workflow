<step id="1" name="load_jobs">

<step id="2" name="qualify_jobs">
For each job, read the `description` column and assess qualification using this rule:

<qualification_rule>
Extract the required years of experience from the job description.
The candidate has YOUR_YEARS_EXP years. Compare strictly:
  - Required <= YOUR_YEARS_EXP years                        -> QUALIFIED
  - Required YOUR_YEARS_EXP+1 to YOUR_YEARS_EXP+2 years    -> BORDERLINE (tailor with note)
  - Required YOUR_YEARS_EXP+3 or more years                 -> SKIP
  - No years mentioned                                      -> QUALIFIED (assume senior-level fit)

Seniority titles:
  - "Staff", "Principal", "Distinguished", "Fellow" in the title -> SKIP
  - "Associate", "Junior", "Entry-level" in the title -> SKIP (overqualified/underleveled)
  - "Senior" or unspecified -> QUALIFIED

Skills qualification:
  Cross-reference the JD's required skills against the base CV and candidate_context.md.
  - If the JD's REQUIRED skills (not nice-to-haves) are substantially covered -> QUALIFIED
  - If 1-2 required skills are missing but the rest of the profile is strong -> BORDERLINE
  - If the core required skills are absent from both source files -> SKIP
    Example: a role that requires primarily C++, embedded systems, or any domain stack
    not present in either file should be skipped.
  Set notes to: "Skipped: core required skills not in profile ([missing skills])"
  or: "Borderline: missing [skill(s)] but strong match otherwise"
</qualification_rule>

Update the CSV for SKIPPED jobs immediately:
- Set `status` = "skipped"
- Set `notes` = "Skipped: [reason]"
  Example: "Skipped: requires 10 years of experience"
  Example: "Skipped: Associate/Junior level role"

Do NOT tailor resumes for skipped jobs. Move to the next job.
</step>

<step id="3" name="tailor_resume">

<substep id="3b" name="tailoring_rules">

<title_rules>
<!--
SETUP: update the title variants below to match your own target title(s).
The rule is: pick the variant that best fits the JD's emphasis. Never invent
a title not listed here, and never change the seniority level.
-->
- NEVER change the seniority level of the title (e.g. never drop "Senior" if the base CV uses it).
- The title may be adjusted between these variants based on the JD's emphasis:
    "YOUR_TITLE_VARIANT_1"
    "YOUR_TITLE_VARIANT_2"
- Apply the same title adjustment to the most recent job title in the experience section. It should match the CV header title.
- Never use a title that introduces a different seniority level.
</title_rules>

<summary_rules>
- Rewrite the summary to mirror the JD's language and priorities.
- Ground every claim in something real from the base CV or candidate_context.md.
- Connect transferable strengths (distributed systems, data-driven platforms, product
  collaboration, LLM tooling) to the role's needs without inventing domain experience.

<voice_rules>
Write in first-person implicit voice - the same way a person naturally writes
a resume summary. This means:

- No "Brings", "Offers", "Delivers", "Drives", "Owns" as opening verbs for a
  clause. These are third-person framings. A person writes "7 years building..."
  not "Brings 7 years of experience building..."
- No sentences that address the reader or reference the job description.
  WRONG: "Experience with X translates directly to this role."
  WRONG: "Hands-on with the exact stack this role requires."
  RIGHT: Just describe the experience. The reader draws the connection.
- No closing sentences that summarize your own qualities in the abstract.
  WRONG: "Strong software architecture and written English communication skills."
  WRONG: "Collaborative, hands-on approach with a track record of..."
  RIGHT: If a quality matters, surface it through a concrete fact, not a label.
- After writing the summary, read each sentence and ask: would a real person
  write this about themselves? If it sounds like a recruiter wrote it about a
  candidate, rewrite it.
</voice_rules>

DOMAIN LANGUAGE (STRICT):
- Do NOT name the JD's industry, domain, or product category in the summary.
  WRONG: "directly applicable to financial and investment management systems"
  WRONG: "directly applicable to building operational infrastructure for airspace management"
  RIGHT: describe platform properties (scale, correctness, real-time, data-driven)
         without naming the domain. The hiring manager knows their domain.
- Exception: if the JD's industry matches one listed in candidate_context.md, you MAY name it. Otherwise never.

- You MUST NOT claim domain expertise or background the candidate does not have.

- The summary must read like something a human would write about themselves, not
  like copy written about a person.

<summary_example>
BAD - avoid this style:
"Actively uses AI development tools as a core part of the engineering workflow -
not as an experiment, but as a daily practice to ship better software faster."

GOOD - aim for this style:
"N years building distributed backend systems at scale, with recent focus on
LLM infrastructure and event-driven architecture. Integrates AI development
tooling into daily engineering practice."

BAD - third-person marketing voice:
"Brings a strong ownership mindset and a proven track record of delivering
resilient infrastructure at scale."

GOOD - implicit first-person:
"Track record of owning infrastructure end-to-end, from system design through
production monitoring."
</summary_example>
</summary_rules>

<skills_rules>
- Reorder skills to lead with those most relevant to the JD.
- You MAY add adjacent skills from the <skills_adjacency_list> in candidate_context.md -
  skills the candidate is legitimately close to and can speak to in an interview.
- Do NOT add skills not present in that list, the base CV, or candidate_context.md.
- Do NOT add domain knowledge the candidate does not have (fintech compliance, medical devices, etc.)
</skills_rules>

<experience_rules>
- Reorder bullets within each role to lead with the most relevant achievements.
- Reframe bullet language to echo the JD's terminology where truthful.
  Example: if the JD says "observability", and a bullet mentions "monitoring dashboards",
  reframe it as "implemented observability infrastructure..."
- Do NOT invent new bullets or new achievements.
- Write a maximum of 4 bullets per job. If the base CV has 5, drop the least relevant one for this JD.
- No bullet may exceed 180 characters.
- If you write a bullet over 140 characters, balance it with at least one bullet under 120 characters
  in the same job. A long bullet is fine; all bullets being long is not.
</experience_rules>

<formatting_rules>
- Never use em dashes (—) anywhere in the output. Use a regular hyphen (-) instead.
  This applies to bullets, summaries, titles, notes, and all other text.
- Formatting and layout are handled entirely by render-cv.js. Do not include any CSS or HTML.
</formatting_rules>

<csv_status_rules>

For BORDERLINE jobs (tailored despite slight underqualification):

- Set `notes` = "Borderline: requires [X] years. Tailored for: [angle summary]"

For SKIPPED jobs:
- Set `status` = "skipped"
- Set `notes` = "Skipped: [reason]"

For jobs where PDF rendering failed:
- Set `status` = "tailor_error"
- Set `notes` = "Error: [brief description of what failed]"
